# Repository Guidelines

Trail Whisper is a React 18 + TypeScript single-page app (Vite build) that uses
Supabase for auth, storage and PostGIS geospatial queries. It parses FIT files
client-side and tells you whether you have already visited your current
location.

## Project Structure & Module Organization
- `src/main.tsx` is the entry point; `src/App.tsx` defines the routes and the
  `ProtectedRoute`/`PublicOnlyRoute` auth gates. Global styles live in
  `src/index.css` (Tailwind).
- `src/routes/` holds page components: `Login`, `Dashboard` (`/app`), `Upload`,
  `History`, `ActivityDetail` (`/activity/:id`), `Account`.
- `src/components/` holds shared UI, grouped by area: `layout/`, `dashboard/`,
  `location/`, `maps/`, and shadcn-inspired primitives in `components/ui/`.
- `src/hooks/` contains data/query hooks (`useActivities`, `useActivityDetail`,
  `useVisitsNear`, `useGeolocation`, `useReverseGeocode`).
- `src/lib/` holds cross-cutting modules: `supabase.ts` (client),
  `fit.ts` (FIT parsing â†’ WKT), `types.ts`, `utils.ts`.
- `src/utils/` holds pure helpers (`geo.ts` Haversine/WKB, `format.ts`).
- `src/providers/AuthProvider.tsx` exposes the `useAuth` context.
- `supabase/setup.sql` provisions the `activities` table, RLS policies and the
  `find_visits_near` / `get_activity_geojson` RPCs. Re-run it after schema edits.
- `public/` holds static assets copied verbatim (favicons, manifest).
- `dist/` is generated by `npm run build` and stays untracked.
- Import alias: `@/` maps to `src/` (configured in `vite.config.ts` and
  `tsconfig.json`).

## Build, Test, and Development Commands
- `npm install` installs dependencies; run after `package.json` /
  `package-lock.json` changes.
- `npm run dev` starts the Vite dev server at `http://localhost:5173` with HMR.
- `npm run build` produces the production build in `dist/` (base path
  `/trail-whisper/` for GitHub Pages).
- `npm run preview` serves the production bundle locally for smoke testing.
- `npm run lint` runs ESLint (flat config, `eslint.config.js`) over `ts`/`tsx`
  files; keep it clean before pushing.
- `npm run typecheck` runs `tsc --noEmit` (the build itself does not type-check).
- `npm test` runs the Vitest unit/component suite; `npm run test:watch` for
  watch mode; `npm run test:e2e` runs the Playwright smoke suite.
- Env vars are required at build time: `VITE_SUPABASE_URL`,
  `VITE_SUPABASE_ANON_KEY`, `VITE_EMAIL_REDIRECT_URL`. Keep them in a local
  `.env` / `.env.local` (gitignored); never commit secrets.

## Coding Style & Naming Conventions
- TypeScript with `strict` mode, modern ES modules, functional React
  components, 2-space indentation, no semicolons (match existing files).
- Single quotes for strings and imports; double quotes for JSX attributes.
- PascalCase for components, `use`-prefixed camelCase for hooks, kebab-case for
  files in `components/ui`.
- Use the `@/` alias for cross-folder imports rather than long relative paths.
- Tailwind utility classes for styling; use the `cn` helper from
  `@/lib/utils` to compose conditional class names.

## Testing Guidelines
- Vitest + Testing Library (jsdom). Place specs next to the unit as
  `Name.test.ts(x)`; pure logic, component smoke tests, and a FIT-binary
  fixture (`src/test/fit-fixture.ts`) exercising the real `fit-file-parser`.
- Playwright smoke suite in `e2e/` runs offline (Supabase env stubbed; with no
  session the app redirects to `/login`).
- Run `npm run lint`, `npm run typecheck`, `npm test`, and `npm run test:e2e`
  before pushing; all must be green.
- Still manually smoke-test map/upload flows via `npm run preview` when touching
  geolocation, FIT parsing, or the map (no automated WebGL coverage): sign in on
  `/login`, accept geolocation, upload a FIT file on `/upload`, verify it
  appears in `/history` and the dashboard, open `/activity/:id`.
- Document manual test notes in the PR description.

## Commit & Pull Request Guidelines
- Follow the existing history: concise, imperative summaries ("Add hero
  section"), optionally referencing issues (`#12`).
- Ensure `npm run lint` and `npm run build` pass locally before opening a PR.
- PRs should describe the change, note testing performed, and call out any
  user-facing or deployment impact.

## Deployment Notes
- `.github/workflows/deploy.yml` builds on every push to `main` (Node 20) and
  deploys to GitHub Pages. `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY`
  must be configured as GitHub Actions secrets.
- Use topic branches for work-in-progress to avoid unintended deployments.
